
# ТЗ: Модуль контроля, отмены визитов, заметок и системы Аудита (Zeus Auto)

### 📌 Общая концепция

Внедрение системы сквозного контроля действий персонала (Аудит) для ролей с уровнем доступа `Owner`. Задача — логировать любые изменения финансовых или статусных данных визитов, дать возможность добавлять рабочие заметки к автомобилям без раздувания интерфейса таблицы, а также безопасно отображать отмененные заказы.

### 🗂️ 1. Изменения в навигации (Сайдбар)

- **Новый компонент:** В левый вертикальный сайдбар добавляется новая вкладка **«📋 Аудит»** (разместить в самом низу списка, под «Архивом смен»).
    
- **Ограничение доступа:** Настроить жесткую проверку прав при рендере сайдбара. Пункт меню «Аудит» отображается **только** если `user.role === 'owner'`. Для базовой роли `admin` этот пункт физически вырезается из DOM-дерева.
    

### 🎨 2. Лента визитов (Интерфейс Dashboardadmin.png)

#### А. Компонент Заметок (Рядом с Маркой автомобиля)

- **Логика отображения:** Если в объекте визита поле `note` не пустое, то в первой колонке таблицы, строго справа от названия машины (например, `Камри`), рендерится компактный жёлтый тег **`+1 заметка`** (использовать аналогичный класс и стили, что и для тега `+1 доп` в колонке услуг).
    
- **Интерактив:** При наведении курсора (`hover`) на тег `+1 заметка`, срабатывает всплывающая подсказка (`Tooltip`), внутри которой отображается текстовое содержимое заметки. Если заметки нет — тег не рендерится.
    

#### Б. Визуальный стиль Отмененных визитов

- Отмененные визиты не удаляются из текущей ленты смены автоматически.
    
- Для строки таблицы (`tr`) отмененного визита принудительно применяется класс со стилями **`opacity: 0.3`**, цвет текста меняется на приглушенно-серый, а неоновые обводки элементов гаснут. Визит визуально уходит на задний план.
    

#### В. Маркер отредактированных данных

- Если визит имеет историю изменений стоимости или услуги, в колонке «Стоимость» рядом с ценой загорается тусклый оранжевый индикатор (точка-badge).
    
- При наведении на этот индикатор выводится `Tooltip` с историей в формате: _«Было [Старая цена] ➔ Стало [Новая цена] (Имя админа, Время)»_.
    

### 🔄 3. Интерфейсы взаимодействия (Модальные окна)

#### А. Окно «Редактировать визит» (Кнопка «Карандаш»)

- В существующее рабочее модальное окно редактирования визита добавить новое текстовое поле ввода: **«Заметка к визиту»** (`textarea`).
    
- При отправке формы данные из этого поля перезаписывают значение `note` в строке визита базы данных.
    

#### Б. Окно «Отмена визита» (Кнопка «Крестик»)

- При клике на «Крестик» в колонке «Действия» открывается модальное окно подтверждения отмены.
    
- Внутри окна разместить обязательное текстовое поле: **«Укажите причину отмены визита»**.
    
- Кнопка подтверждения «Удалить/Отменить визит» должна находиться в состоянии **`disabled`**, пока поле причины пустое или содержит менее 5 символов.
    

### ⚙️ 4. Логика Бэкенда и Базы Данных (Таблица Аудита)

#### А. Новая таблица в PostgreSQL: `action_logs`

Необходимо создать таблицу для хранения логов аудита со следующей структурой:

- `id` (Primary Key, UUID)
    
- `timestamp` (DateTime, по умолчанию `NOW()`)
    
- `admin_id` (Foreign Key, указывает на админа, совершившего действие)
    
- `visit_id` (Foreign Key, указывает на измененный визит)
    
- `action_type` (Enum: `'edit'`, `'cancel'`, `'note_add'`)
    
- `payload` (JSONB — для хранения деталей изменений, например: `{"old_price": 3100, "new_price": 2500, "reason": "текст причины"}`)
    

#### Б. Бизнес-логика эндпоинтов

1. **При отмене визита (`POST /api/visits/:id/cancel`):**
    
    - Статус визита в таблице `visits` меняется на `'cancelled'`.
        
    - Автоматически создается запись в `action_logs` с типом `'cancel'`, куда записывается `admin_id` из сессии/токена и переданная из модального окна причина отмены.
        
2. **При редактировании визита (`PUT /api/visits/:id`):**
    
    - Бэкенд сравнивает пришедшие данные с текущими в БД.
        
    - Если изменилась стоимость или услуга, бэкенд обновляет визит и параллельно делает запись в `action_logs` с типом `'edit'`, сохраняя старые и новые значения в JSON-payload.
        
3. **Безопасность таблиц:** Для таблицы `action_logs` запрещены любые операции `DELETE` и `UPDATE` для всех ролей, включая `Owner`. Данные туда могут только записываться системой.
    

### 📋 5. Экран «Аудит» (Внутри новой вкладки сайдбара)

- Компонент представляет собой вертикальную ленту активности (`Activity Feed`), отсортированную по `timestamp` от самых свежих событий к старым.
    
- Каждая строка ленты — это карточка лога: **[Время] | [Имя сотрудника] | [Действие (Цветовой маркер типа)] | [Детали из payload]**.
