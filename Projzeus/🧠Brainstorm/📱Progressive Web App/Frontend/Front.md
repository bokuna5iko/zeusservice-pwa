# 🎨 Frontend (Визуальная часть и интерфейс)

Этот файл описывает внешний вид PWA, стилистику и структуру экранов.

## 🌈 Дизайн-система (UI Kit)
Чтобы приложение выглядело цельно, придерживаемся следующих параметров:

- **Цветовая палитра:**
  - `Primary (Main):` [Например, Темно-синий/Черный] — для фона.
  - `Accent:` [Например, Ярко-зеленый или Золотой] — для активных карточек и кнопки "Бесплатная мойка".
  - `Text:` Белый/Светло-серый (для контраста с темным фоном).
- **Шрифты:** Без засечек (Sans-serif), читабельные на ходу.

## 📱 Структура экранов (Layout)

### 1. Главный экран (Dashboard)
- **Счетчик посещений:** Визуальные иконки (например, 8 пустых кружков/машинок, которые заполняются цветом по мере визитов).
- **Инфо-блок:** "Осталось [N] моек до подарка".

### 2. Экран поздравления (Loyalty Popup)
- Появляется, когда `counter == 8`.
- **Элементы:** Большая надпись "ПОЗДРАВЛЯЕМ!", яркая подсветка, кнопка активации.

## 🏗 Верстка и CSS
- **Адаптивность:** Mobile First! (Приложение должно идеально выглядеть на iPhone и Android).
- **Flexbox/Grid:** Используем для центрирования карточек и макета.
- **Анимации:** - Плавный переход (transition) при заполнении счетчика.

## 🖼 Ассеты (Графика)

- Иконки PWA: `/assets/icons/`.

## ТЗ на разработку модуля «Интерактивный прейскурант»

### 1. Общие требования к интерфейсу

- **Компонент:** Модальное окно (Modal Overlay).
    
- **Цветовая схема:** Темный фон (Deep Navy/Black), белый и золотистый/серебристый акценты (согласно логотипу Zeus Auto).
    
- **Шрифты:** Без засечек (например, Inter или Roboto) для высокой читаемости на мобильных устройствах.
    

### 2. Функциональные блоки модального окна

#### 2.1 Шапка (Header)

- Заголовок: «Прейскурант услуг».
    
- Кнопка закрытия (крестик) в верхнем правом углу.
    
- **Селектор классов (Sticky Bar):** Горизонтальный переключатель (Tabs), который фиксируется наверху при скролле.
    
    - Элементы: `1 класс`, `2 класс`, `3 класс`, `4 класс`, `5 класс`.
        
    - _Поведение:_ По умолчанию выбран «1 класс». При смене класса цены в списке ниже обновляются мгновенно (без перезагрузки окна).
        

#### 2.2 Список услуг (Content)

Услуги делятся на две логические группы:

**Группа №1: Услуги с градацией по классам**

- _Список:_ 2-х фазная мойка, Стандартная мойка + коврики, Комплексная мойка, Мойка днища.
    
- _Отображение:_ Название услуги (крупно) + Цена (ярко).
    
- _Логика цены:_ Берется из массива данных в зависимости от активного таба в Header.
    

**Группа №2: Услуги с фиксированной ценой**

- _Список:_ Все остальные (Мойка мотоцикла, двигателя, химчистка, пылесос и т.д.).
    
- _Отображение:_ Название + Цена (например, "от 500 ₽"). Цена не меняется при переключении классов.
    

#### 2.3 Система описаний (Details)

- Использовать механизм **Accordion (Аккордеон)**.
    
- Рядом с названием услуги (или под ним) находится иконка стрелочки или «i».
    
- При клике на строку услуги раскрывается блок с подробностями (тест из прайса).
    
    - _Пример для 2-х фазной мойки:_ «Мойка эмульсией, мойка пенным кислотным шампунем, полная сушка, чернение резины».

**Общая концепция:** Экран администратора совпадает по табам с клиентским, но содержит рабочий функционал для учета, сканирования и контроля плана смены.

### Контейнер №1: План на смену (Progress Bar)

- **Расположение:** Самый верх экрана.
    
- **Интерфейс (UI):** * Линейный прогресс-бар с динамическим курсором, отображающим текущее количество помытых машин за сегодня в реальном времени.
    
    - На шкале установлены 4 контрольные точки (метки): **30**, **60** (целевой план от Артема), **75** и **90**.
        
    - Метки **75** и **90** визуально выделены (например, другим цветом), сигнализируя о перевыполнении плана.
        
- **Логика/Бэкенд:** При загрузке экрана отправляется запрос на получение общего количества визитов за текущие сутки 
  Пример: (`GET /api/admin/stats/today-count`). 
    

---

### Контейнер №2: Кнопка сканирования QR и Калькулятор

- **Расположение:** Чуть ниже середины экрана (удобная зона под большой палец).
    
- **Интерфейс кнопки:** Крупный квадратный блок с иконкой QR-сканера. По клику запускается камера.
    
- **Модальное окно «Калькулятор» (открывается после успешного сканирования):**
    
    - **Логика лояльности (`visit_count`):** Система проверяет счетчик визитов клиента.
        
        - _Обычный визит (не 4-й и не 8-й):_ При выборе услуги из выпадающего списка подтягивается базовая цена.
            
        - _4-й визит (Скидка 20%):_ В интерфейсе дублируются две цены: дефолтная (зачеркнутая) и цена со скидкой 20% (активная).
            
        - _8-й визит (Бесплатно):_ В интерфейсе дублируются две цены: дефолтная (зачеркнутая) и цена 0 рублей.
            
        - _Ручной ввод:_ Если цена услуги нестандартная, должна быть возможность вбить сумму вручную в отдельное инпут-поле.
        - Выбор услуги: Подтягиваем в нашей bd таблицу services и подтягиваем id каждого service_name c base_price.
          Благодаря подтягиванию, администратор сможет выбрать услугу которую взял клиент 
            
    - **Учет оплаты:** Перед отправкой админ выбирает тип оплаты через переключатель (радио-кнопки): **«Наличные»** или **«Онлайн-перевод»**
        
- **Действие:** По нажатию кнопки «Зачислить визит» отправляется `POST`-запрос на бэкенд, обновляющий баланс клиента и добавляющий запись в историю как администратору за зачисление визита, так и клиенту в историю о прошедшем визите. Интерфейс выдает успешный микро-отклик (зеленая галочка/чек-ап), и модалка закрывается.

- **Логика:** Сценарий для клиентов без приложения. По клику открывается то же модальное окно «Калькулятор», но с флагом `is_guest: true`.
    
- **Техническое требование к БД:** Данный визит записывается в таблицу учета без привязки к конкретному `client_id` (ставим `NULL`), но со всеми данными: выбранная услуга, итоговая стоимость, тип оплаты и дата. Номер телефона не запрашивается, профиль в базе не создается.

# ТЗ: Экран «История» для аккаунта Администратора (PWA zeus-auto)

**Общая концепция:** Экран отображает ленту всех посещений за последнюю неделю с возможностью сквозного поиска. Стилистика, структура карточек и анимация «гармошки» (аккордеона) полностью наследуются из компонента клиентской истории для сохранения единого UI-кита.

### Контейнер №1: Панель фильтрации и поиска (Smart Search)

- **Интерфейс (UI):** Единое прямоугольное инпут-поле (Поиск) в общей темной стилистике приложения.

### Контейнер №2: Лента визитов (Аккордеон)

- **Расположение:** Основная часть экрана под поисковой строкой.
    
- **Интерфейс (UI):** Полностью копирует CSS-классы, разметку и стили компонента `History` из клиентского аккаунта.
    

#### Сценарий №1: Запись обычного клиента (Зарегистрирован по QR)

- **В закрытом состоянии (Превью карточки):**
    
    - Время визита (`visits.created_at` форматированное в `HH:MM`).
        
    - Тип услуги (`visits.service_type`).
        
    - Марка машины из профиля пользователя (`users.car_brand`).
        
- **В открытом состоянии (Гармошка раскрыта):** По клику на карточку плавно выезжает доп. информация о клиенте из таблицы `users`:
    
    - ID клиента, ФИО (`name`), телефон (`phone`).
        
    - Роль (`role`), Всего визитов (`total_visits`), Счетчик текущей карты лояльности (`visit_count`).
        

#### Сценарий №2: Запись гостя (Добавлен без QR по кнопке «Гость»)

- **В закрытом состоянии:** Время, тип услуги, а вместо марки машины пишется плашка: **«Гость (Без приложения)»**.
    
- **В открытом состоянии:** Поля профиля (`name`, `phone`, `visit_count`) скрываются или заменяются заглушкой: _«Данные профиля недоступны для гостевых визитов»_. Приложение не должно падать при попытке прочитать `NULL` из базы.
    

### Техническая подсказка для Бэкенда (Node.js + PostgreSQL)

Для реализации этого экрана кодеру потребуется составить `LEFT JOIN` запрос. Вот готовый SQL-шаблон, который нужно использовать в эндпоинте `GET /api/admin/history`:

SQL

```
SELECT 
    v.id AS visit_id, 
    v.created_at, 
    v.service_type AS service_name, -- приводим к единому имени для фронта
    v.price, 
    v.payment_type,
    v.visit_number,
    u.id AS user_id, 
    u.name, 
    u.phone, 
    u.car_brand, 
    u.role, 
    u.total_visits, 
    u.visit_count
FROM visits v
LEFT JOIN users u ON v.user_id = u.id
WHERE v.created_at >= NOW() - INTERVAL '7 days'
ORDER BY v.created_at DESC;
```

_Примечание для кодера:_ `LEFT JOIN` обязателен, чтобы визиты со значением `client_id = NULL` (гости) тоже выводились в список.
