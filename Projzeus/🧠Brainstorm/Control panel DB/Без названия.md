
# ТЕХНИЧЕСКОЕ ЗАДАНИЕ (ТЗ)
## Архитектура системы управления данными «Zeus Auto»

### 🔗 Общая концепция
База данных **PostgreSQL** является единым и неделимым источником правды для всего проекта. Над ней разворачиваются два независимых интерфейса управления с четким разделением обязанностей:

1. **Малый пульт (PWA):** Локальная админка на ПК автомойки. Обладает полной автономностью и сохраняет работоспособность кассы даже при полном отключении интернета.
2. **Большой пульт (NocoDB):** Глобальный онлайн-контроллер базы данных в реальном времени. Предназначен для **Босса** и верхнеуровневого менеджмента.

## 📱 БЛОК 1. Малый пульт (Интерфейс администратора в PWA с поддержкой Offline)

Этот блок описывает логику работы фронтенд и бэкенд части PWA для обеспечения бесперебойной работы кассы при падении интернета.

### 1.1. Инфраструктура локального хранения (Браузер кассы)
* **Технология:** Использовать встроенную базу данных браузера `IndexedDB` через библиотеку-обертку (рекомендуется `Dexie.js` для стабильности).
* **Кэширование данных (Online):** * Пока интернет присутствует, PWA при инициализации рабочего дня и затем каждые 15 минут в фоновом режиме запрашивает с бэкенда актуальные срезы данных за текущие сутки (список визитов, активные смены, справочник услуг и цен, список зарегистрированных пользователей).
  * Данные перезаписываются в `IndexedDB`, формируя актуальный «слепок» утренней и дневной кассы на жестком диске ПК кассира.

### 1.2. Поведение интерфейса при потере сети (Offline-режим)
* **Перехват запросов:** `Service Worker` (через Workbox) при фиксации ошибки сети (`NetworkError`) переключает стратегию получения данных с `NetworkOnly` (или `NetworkFirst`) на `CacheOnly` для административных маршрутов.
* **UI/UX Индикация:** * На экране PWA администратора должна динамически появляться фиксированная верхняя плашка (Sticky Banner) желтого/красного цвета: 
    > ⚠️ Режим ЧП: Интернет отсутствует. Новые записи (Х) сохраняются локально.
  * При этом интерфейс списков (утренние визиты, цены) остается полностью доступным для чтения, данные берутся напрямую из локальной `IndexedDB`.
* **Добавление/Редактирование данных в Offline:**
  * При нажатии кнопки «Добавить визит» или «Изменить смену», PWA не отправляет стандартный `POST/PUT` запрос на сервер, а валидирует поля локально и записывает новую строку в оффлайн-очередь (`Outbox Queue`) внутри `IndexedDB`.
  * Пользователю выводится нативный UI-нотификатор: *«Запись успешно сохранена в локальную память устройства»*.

### 1.3. Логика бесконфликтной пакетной синхронизации
* **Формат Идентификаторов (Критично):** Во всех таблицах базы данных (`users`, `visits`, `shifts`, `services`) в качестве первичных ключей (`id`) вместо инкрементных чисел (1, 2, 3...) **обязательно использовать UUIDv4**. Это позволит PWA генерировать гарантированно уникальные ID в оффлайне без риска совпадения с записями, созданными в этот же момент в онлайне.
* **Автоматический триггер:** В код PWA интегрируется слушатель событий сети:
 ```javascript
  window.addEventListener('online', syncOfflineData);
 ```

- **Пакетная выгрузка:** Как только событие `online` сработало, PWA блокирует интерфейс на 1–2 секунды, считывает все накопившиеся записи из `Outbox Queue` в `IndexedDB` и отправляет их на бэкенд единым JSON-пакетом (один массив данных на специальный эндпоинт, например `/api/admin/sync-offline`).
    
- **Обработка на Бэкенде (Node.js):** Бэкенд принимает пакет, в рамках одной SQL-транзакции производит массовую запись (`UPSERT` или `INSERT ... ON CONFLICT DO NOTHING`) в PostgreSQL. После успешного ответа сервера PWA очищает `Outbox Queue`, а плашка ЧП меняется на зеленый цвет:
    
    > ✅ Все данные успешно синхронизированы с базой.
    

## 🖥️ БЛОК 2. Большой пульт (Глобальный контроллер NocoDB)

Этот блок описывает настройку No-Code платформы NocoDB в качестве безопасной, продвинутой СУБД-панели для Босса, работающей в режиме реального времени.

### 2.1. Подключение и Синхронизация с PostgreSQL

- **Режим интеграции:** NocoDB подключается к существующей продакшн-базе PostgreSQL как внешний `Data Source`. Режим создания изолированной локальной базы NocoDB запрещен.
    
- **Реактивность:** Любое изменение ячейки в NocoDB (например, Босс зашел со смартфона и изменил стоимость услуги в таблице `services`) должно мгновенно триггерить SQL-запрос `UPDATE` в PostgreSQL, чтобы клиенты в PWA сразу увидели новую цену.
    

### 2.2. Защита от дурака (Валидация, Маскирование и UI-ограничения)

- **Типизация колонок:** В интерфейсе NocoDB для каждой таблицы жестко настраиваются типы отображения (Column Views):
    
    - Поля _«Цена»_, _«Баллы»_ — тип `Number` или `Currency` (блокировать ввод любых буквенных символов на уровне интерфейса).
        
    - Поля _«Телефон»_ — тип `Phone` или текстовое поле с Regex-валидацией по международной маске.
        
- **Реализация связей (Relations через выпадающие списки):** * Поля внешних ключей (например, `user_id` в таблице визитов) преобразуются в тип связи `Many-to-One` / `Belongs To`.
    
    - При ручном добавлении визита Боссом, вместо ввода системного UUID, NocoDB должен открывать выпадающий список (Dropdown) с удобным поиском по ФИО/Телефону из связанной таблицы `users`.
        

### 2.3. Маскирование системных данных и уровни доступа

- **Скрытие технических колонок:** Для предотвращения случайного удаления или повреждения логики приложения, из стандартного табличного вида NocoDB скрываются (`Hide Column`) или переводятся в режим `Read-Only` следующие поля:
    
    - Первичные и внешние ключи (`id`, `uuid`, `user_id`, `service_id`).
        
    - Системные временные метки (`created_at`, `updated_at`).
        
    - Хэши паролей пользователей (`password_hash`).
        
- **Ролевая модель:**
    
    - **Учетная запись Босса:** Роль `Owner` / `Creator`. Полный доступ к изменению цен, управлению структурой, просмотру всей аналитики, логов и смен.
        
    - **Учетные записи Администраторов/Наблюдателей** (при необходимости выдачи доступа в NocoDB): Роль `Editor` (с ограничением видимости таблиц) или `Viewer` (строго только чтение для мониторинга без права изменять ячейки).
        

### 2.4. Аудит-логирование (История изменений)

- **Активация модуля:** В настройках проекта NocoDB включается встроенный компонент **Audit Logs**.
    
- **Требования к логам:** Каждое ручное изменение, сделанное внутри интерфейса NocoDB, должно логироваться с автоматической фиксацией:
    
    - _Кто изменил (Email/User)_
        
    - _Когда (Timestamp)_
        
    - _Какая таблица / строка_
        
    - _Старое значение ➡️ Новое значение_
        
- Кнопка отката изменений (`Rollback`) должна быть доступна Боссу в один клик через боковую панель истории конкретной строки.
