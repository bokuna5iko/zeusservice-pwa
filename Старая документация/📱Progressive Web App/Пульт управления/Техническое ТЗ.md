
# 🛠 Техническое ТЗ: Архитектурный план интеграции «Пульта управления»

## 🧱 БЛОК 1: Подготовка (Работа с БД и логика сущностей)

Задача блока — заложить архитектурную концепцию сменности, к которой бэкенд будет привязывать все финансовые и операционные события дня.

- **Логика Смен:** Создать логическую сущность «Смена». Она должна фиксировать календарную дату, статус (открыта/закрыта), временные метки начала/окончания рабочего дня и агрегировать в себе финансовые итоги (общие суммы наличных, безналичных и расходов строго внутри этой сессии).
    
- **Логика Расходов:** Добавить сущность для учета мелких трат в моменте. Каждая запись жестко привязывается к текущей открытой смене.
    
- **Связь с Визитами:** Связать сущность визитов с текущей открытой сменой. Визит должен уметь принимать и хранить плоские текстовые параметры для оперативного ручного ввода админом: марку машины, контактные данные клиента, вид услуги, итоговую стоимость и выбранный способ расчета.
    
- **Счетчик акций:** Предусмотреть вывод в визит текущего шага клиента по карте лояльности (число от 1 до 8), который транслируется из существующих таблиц пользователей.
    

## 🎨 БЛОК 2: Создание Фронтенд-части (React)

Задача блока — реализовать адаптивный запуск интерфейса в рамках единого приложения и настроить трехсекционный хаб для ПК.

- **Логика адаптивного роутинга:** * Проверка прав: доступ строго при наличии роли администратора.
    
    - Разделение по экрану: при ширине экрана `< 1024px` (смартфоны) монтируется стандартный мобильный интерфейс админки PWA. При ширине `>= 1024px` (ПК/планшеты) приложение автоматически переключает админа на десктопный полноэкранный «Пульт управления».
        
- **Реализация Вкладок (Табов):**
    
    - **Вкладка 1: Лента визитов:** Закрепленная верхняя панель с кассовыми виджетами и кнопкой добавления трат. Основную площадь занимает плотная интерактивная таблица из 8 колонок в строго утвержденном порядке (Номер, Марка, Имя+Username, Телефон, Услуга, Цена, Номер в акции, Оплата) с быстрыми кнопками изменения и удаления в конце строки.
        
    - **Вкладка 2: Работа с сотрудниками:** Экран со списком персонала, возможностью отметки их явки на смену и выводом их индивидуальной выработки/процента за день.
        
    - **Вкладка 3: Мобильная админка (Симулятор):** Окно-фрейм в форме смартфона, внутри которого переиспользуются компоненты мобильной версии админки. Позволяет админу совершать «телефонные» действия, не вставая из-за ПК.
        
- **Умные формы:** Переиспользуемое модальное окно для создания/редактирования записей с автодополнением полей при вводе и выпадающими списками пресетов.
    

## ⚙️ БЛОК 3: Написание Бэкенда и Логики Реального Времени

Задача блока — обеспечить точечное сохранение данных и настроить мгновенный обмен событиями между клиентами и пультом.

- **Управление жизненным циклом смены:** Реализовать эндпоинты для открытия дня в 09:00 (с проверкой уникальности даты) и закрытия в 22:00 (с блокировкой дальнейших изменений и расчетом финального баланса кассы).
    
- **Точечное обновление (PATCH):** Любое исправление админа в таблице «в моменте» отправляет мгновенный точечный запрос на сервер для перезаписи конкретных полей конкретного визита. Это исключает потерю данных.
    
- **Мгновенная синхронизация через WebSockets (Socket.io):**
    
    - Пульт админа держит постоянное сокет-соединение с сервером и подписывается на административную комнату.
        
    - Как только клиент совершает целевое действие в PWA (сканирует QR-код, создает визит), бэкенд фиксирует это в БД и мгновенно отправляет сокет-событие на пульт.
        
    - Фронтенд админа ловит сокет-событие и «на лету» модифицирует массив данных (добавляет строку, обновляет статус или удаляет запись) без перезагрузки страницы и без использования таймеров опроса.
